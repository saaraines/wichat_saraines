[[section-quality-scenarios]]
== Requisitos de Calidad

=== Árbol de Calidad

[options="header",cols="1,2"]
|===
| Característica de Calidad | Aspectos Relevantes
| *Funcionalidad* | Correctitud de respuestas, integración con APIs externas, sistema de pistas funcional
| *Fiabilidad* | Manejo de errores, disponibilidad de servicios, recuperación ante fallos
| *Usabilidad* | Interfaz intuitiva, mensajes claros, experiencia de usuario fluida
| *Eficiencia* | Tiempos de respuesta rápidos, uso optimizado de APIs externas
| *Mantenibilidad* | Código modular, documentación clara, facilidad para añadir nuevas funcionalidades
| *Portabilidad* | Ejecución en diferentes sistemas operativos, fácil instalación
| *Testeabilidad* | Cobertura de tests alta, tests automatizados, aislamiento de componentes
|===

=== Escenarios de Calidad

==== Funcionalidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| F1 | El sistema debe validar correctamente las respuestas del usuario comparándolas con la respuesta correcta almacenada en la pregunta
| F2 | El sistema de pistas debe proporcionar ayuda relevante sin revelar directamente la respuesta correcta
| F3 | Las preguntas deben generarse correctamente desde Wikidata con imágenes y opciones múltiples válidas
|===

==== Usabilidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| U1 | Un usuario nuevo debe poder registrarse y comenzar a jugar en menos de 2 minutos
| U2 | Los mensajes de error deben ser claros y en español, indicando qué acción tomar
| U3 | La interfaz debe ser responsive y funcionar correctamente en dispositivos móviles y desktop
| U4 | El chat de pistas debe mostrar claramente quién habla (usuario o IA) y mantener el historial visible
|===

==== Fiabilidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| R1 | Si Wikidata no responde, el sistema debe usar preguntas previamente almacenadas en MongoDB
| R2 | Si Gemini API falla, el sistema debe informar al usuario que las pistas no están disponibles temporalmente
| R3 | Los servicios deben manejar errores de conexión a MongoDB sin provocar caídas del sistema
|===

==== Eficiencia

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| E1 | Las preguntas deben cargarse en menos de 2 segundos en condiciones normales de red
| E2 | El sistema debe reutilizar preguntas almacenadas para evitar consultas repetidas a Wikidata
| E3 | Las pistas de IA deben generarse en menos de 5 segundos
|===

==== Mantenibilidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| M1 | Añadir una nueva categoría de preguntas debe requerir cambios solo en el Question Service y componentes frontend relacionados
| M2 | Cada microservicio debe poder ejecutarse y probarse de forma independiente
| M3 | El código debe seguir una estructura consistente para facilitar la incorporación de nuevos desarrolladores
|===

==== Testeabilidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| T1 | Todos los servicios backend deben tener tests unitarios con cobertura superior al 80%
| T2 | Los tests E2E deben cubrir los flujos principales: registro, login, juego, estadísticas
| T3 | Los tests deben ejecutarse de forma automatizada y aislada usando mongodb-memory-server
|===

==== Portabilidad

[options="header",cols="1,3"]
|===
| Escenario | Descripción
| P1 | El sistema debe poder ejecutarse en Windows, Linux y macOS con Node.js y MongoDB instalados
| P2 | Las instrucciones de instalación deben permitir a un desarrollador nuevo poner el sistema en marcha en menos de 15 minutos
|===
